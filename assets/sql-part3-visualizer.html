Bonus concept — The invisible engine

The order you write it is not the order it runs.

Before we look at nested queries, one thing that explains everything after it: SQL is written top-down but the database evaluates it in a different order. Understanding this order is what makes subqueries, CTEs, and window functions click.

Written order

What you type, top to bottom
  • 1SELECTpick columns
  • 2FROMwhich tables
  • 3WHEREfilter rows
  • 4GROUP BYcollapse into groups
  • 5HAVINGfilter groups
  • 6ORDER BYsort results
  • 7LIMITcap row count

Logical order

What the database actually runs
  • 1FROMload the rows
  • 2WHEREdrop rows that fail the filter
  • 3GROUP BYcollapse into buckets
  • 4HAVINGdrop buckets that fail the filter
  • 5SELECTcompute output columns
  • 6ORDER BYsort the output
  • 7LIMITreturn top N
This is why you cannot reference a SELECT alias inside WHERE — WHERE runs before SELECT exists. But you can reference it inside ORDER BY, which runs after. Try it on the right.

Part 3 — Nested Logic · Subqueries

A query inside another query.

A subquery is a SELECT statement nested inside another. The inner query runs first (or, if correlated, runs once per outer row), and its result flows into the outer query. Three flavours below, in the order you'll meet them most.

Part 3 — Nested Logic · CTEs

Name it once, use it like a table.

A CTE (Common Table Expression) is a named temporary result defined with WITH name AS (...) at the top of your query. It computes once, gets a name, and the main query references it like any table. Same power as a subquery, vastly more readable once logic stacks up.

Mental model — Reach for the right tool

Which one, and when?

Both can solve the same problem. The question is readability, reuse, and whether your logic needs the outer row's values. Use this as a quick decision flow and a rule-of-thumb cheatsheet.

Decision flow

Start here Do I need to reference the intermediate result more than once in the same query?
Yes → CTE
A subquery would force you to write the same block twice. CTEs let you define once and reference by name anywhere below.
No → keep reading
Either tool works. Let the next question decide.
Next Does the inner logic need values from the outer row?
Yes → Correlated subquery
Only subqueries can reference the outer row. Heads up: fires once per outer row, so watch performance.
No → next question
Next Is the inner logic more than 4–5 lines, or would it confuse someone reading at a glance?
Yes → CTE
Naming the step beats nesting it. "WITH active_clients AS (...)" reads like a sentence.
No → Subquery is fine
Small IN(...) or scalar subqueries are perfectly readable inline.
Special case Is it hierarchical or recursive data (org charts, threads, tree walks)?
Must use recursive CTE
Subqueries cannot recurse. Only WITH RECURSIVE ... can walk a tree.

Rule-of-thumb cheatsheet

"Does this row's column exist in another table?"
Quick existence check — use IN (...) or EXISTS (...) inline.
Subquery
"I need a single value computed from another query."
Scalar subquery in SELECT or WHERE — keep it short.
Subquery
"The filter depends on the current row's values."
Correlated subquery is the only way. Expect it to fire once per outer row.
Correlated
"My query has three distinct logical steps, and the middle step is reused."
Each step becomes a CTE. Reads like an outline — far easier to code-review.
CTE
"I want to compute aggregates, then filter on the aggregate, then join again."
Two CTEs in sequence. Nesting three subqueries is readable by no one.
CTE
"I'm walking a hierarchy — employees, folders, categories."
Only WITH RECURSIVE can self-reference. Subqueries can't.
Recursive CTE
"A quick inline filter — two lines at most."
Subquery. A CTE for something trivial is overkill.
Subquery
"I have to send this query to a teammate in two hours."
Default to CTEs. Named steps age better than nested selects.
CTE
The honest answer in the industry: most analysts default to CTEs for anything beyond trivial, because they age better and review faster. Subqueries are still the right call for tiny inline checks (WHERE id IN (SELECT...)) and for correlated logic (which CTEs can't express). Knowing both means you can read anyone's code.
